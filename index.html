How to use this tool

Definitions, tips, and best practices

Step type definitions
Automated A system or tool does this without staff involvement
Examples: eClinicalWorks auto-sends an appointment confirmation, Waystar runs an eligibility check triggered by a new patient record, the matching algorithm assigns a provider. If staff are copy-pasting or doing the same lookup every single time, it's probably a candidate to automate.
Manual A person must do this — it requires judgment or clinical context
Examples: a care coordinator calls a patient to discuss self-pay options, a clinical supervisor reviews a provider match for a complex case, a psychiatrist decides whether to adjust medication. These steps need a human because the right answer depends on context a system can't evaluate.
Decision A fork — what happens next depends on a condition
Examples: if insurance is out-of-network, route to self-pay discussion; if the patient selects medication management, assign a psychiatrist not a therapist; if preferred provider has no availability within 14 days, offer the next best match. Tag these and write out the rules in the Branching Logic section.
Tips for filling this out well
Name every system explicitly. Don't say "check eligibility" — say "run eligibility check in Waystar." Specific tool names enable better automation recommendations.
Each step = one action, one person or system. "Verify insurance AND create EHR record AND send welcome email" is three steps.
The Pain Points field is the most valuable one. What do coordinators actually complain about? What gets skipped when it's busy? Be honest — that's where the ROI is.
For decision rules, write real "IF / THEN" logic. "Use your judgment" is a gap — try to convert it into an explicit rule anyone could follow.
Include compliance context (NCQA, state licensing rules, payer requirements) in Notes. This flags regulatory risk areas correctly.
